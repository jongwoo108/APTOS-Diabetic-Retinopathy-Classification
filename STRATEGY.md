# 문제 해결 사고 과정
## APTOS 2019 Blindness Detection 사례 기반

---

## Phase 1: 문제 정의 — "무엇을 풀어야 하는가"

### 1-1. 태스크 이해
- 입력: 망막 안저 사진 (Fundus Photography)
- 출력: 당뇨망막병증 중증도 0~4 (5단계)
- 본질: 순서형 분류 (Ordinal Classification) — 단순 분류가 아님
- 의미: 0(정상) → 4(증식성)로 갈수록 심각. 클래스 간 "거리"가 존재

### 1-2. 평가지표 분석
- Quadratic Weighted Kappa (QWK)
- 핵심 특성: 먼 클래스 오분류에 제곱 페널티
  - 0→1 오분류: 페널티 1
  - 0→4 오분류: 페널티 16 (4²)
- 시사점: 정확도(Accuracy)가 아닌 "얼마나 가깝게 틀리느냐"가 중요
- 전략적 판단: 분류보다 회귀 또는 순서형 접근이 유리

### 1-3. 데이터 탐색
- 총 3,662장 (의료 이미지치고는 소규모)
- 클래스 불균형: class 0(1805) vs class 3(193) → 약 9:1
- 시사점: 소수 클래스를 무시하는 모델이 나올 위험 높음
- 관련 도메인 지식: 논문(Abramoff et al. 2010)에서 DR의 단계적 특성,
  fundus 이미지 전처리 방법론, 자동 스크리닝 시스템 선행연구 확인

## Phase 2: 베이스라인 구축 — "먼저 돌아가게 만들기"

### 2-1. 기술 선택 근거
- 모델: EfficientNet-B0 (timm) — 가벼우면서 검증된 아키텍처
- 전처리: Ben Graham 기법 — 2015 DR 대회 우승 기법
- Loss: CrossEntropyLoss — 가장 기본적인 분류 손실
- 목표: 일단 end-to-end 파이프라인을 완성하고 제출까지 도달

### 2-2. 베이스라인 결과
- Validation Kappa: 0.85
- Public Kappa: 0.80
- 분석: 학습은 되지만 최적화 여지가 많음

### 2-3. 베이스라인에서 발견된 문제들
- 코드 버그: classifier 오타로 1000개 출력층 사용
- 전처리 불일치: 학습(0~255 raw) vs 추론(normalize) 불일치
- 환경 차이: Colab과 Kaggle의 timm 버전 차이로 가중치 로드 실패
- 교훈: **먼저 작동하는 코드를 만들고, 그 다음 개선하라**

## Phase 3: 디버깅 — "왜 안 되는지 찾기"

### 3-1. 체계적 진단 프로세스
문제를 세 가지 층위로 분리:

**① 코드 레벨 버그**
- `self.model.classfier` (오타) → `self.model.classifier`여야 함
- forward()가 timm 내부의 1000개 출력층을 통과
- 원인: Python은 오타 속성도 동적으로 생성하므로 에러가 안 남

**② 전처리 불일치**
- 학습 코드의 Dataset.__getitem__을 한 줄씩 추적
- transpose만 하고 /255.0, normalize 없음 → 0~255 범위로 학습됨
- 추론에서 normalize를 넣으면 분포가 완전히 달라짐

**③ 환경 차이**
- Colab timm과 Kaggle timm의 EfficientNet 내부 레이어 구조 상이
- state_dict 키 이름 불일치 → size mismatch 에러

### 3-2. 각 문제의 해결
- ①: 1000개 출력 유지 + forward에서 [:5] 슬라이싱
- ②: 추론에서도 normalize 제거 (학습과 동일하게)
- ③: 유연한 가중치 로더 (suffix 매칭 + shape 검증)
- 결과: 362/362 가중치 100% 매칭, 예측 분포 정상화

### 3-3. 디버깅에서 얻은 교훈
- 모델 성능 문제의 80%는 데이터 파이프라인에 있다
- "학습 코드와 추론 코드의 전처리가 100% 동일한가?"를 항상 첫 번째로 확인
- strict=False로 가중치를 로드하면 문제가 감춰짐 → shape 검증 필수

## Phase 4: 전략적 개선 — "어디를 고치면 가장 효과적인가"

### 4-1. 문제-해결 매핑 (Root Cause → Action)

| 성능 병목 | 근본 원인 | 해결 전략 | 기대 효과 |
|-----------|----------|----------|----------|
| 소수 클래스 무시 | 데이터 불균형 9:1 | WeightedRandomSampler + class weight | 소수 클래스 재현율 개선 |
| 먼 클래스 오분류 | CE Loss가 순서 무시 | CORAL Ordinal Loss | QWK 직접 최적화 |
| 과적합 | augmentation 없음 | Albumentations + ImageNet normalize | 일반화 성능 향상 |
| 표현력 부족 | B0 (5.3M params) | B3 (12M params) + 300px | 미세 병변 감지력 향상 |

### 4-2. 우선순위 결정 기준
- **효과 대비 비용**: 코드 변경량 vs 예상 성능 향상
- **하드웨어 제약**: 무료 Colab T4 15GB
- **리스크**: 학습 시간, 런타임 불안정성
- 결정: 불균형 해소 > Loss 변경 > Augmentation > 모델 업그레이드

### 4-3. 안정성 설계
- 매 epoch 구글 드라이브 체크포인트 저장
- resume 기능으로 런타임 끊김 대응
- gradient clipping으로 학습 안정성 확보
- 교훈: **무료 Colab에서는 "학습이 날아가지 않는 것"이 첫 번째 목표**

## Phase 5: 실행 및 결과 — "가설을 검증하기"

### 5-1. 실험 결과
| 버전 | Validation | Private | Public |
|------|-----------|---------|--------|
| v1 (베이스라인) | 0.85 | 0.800 | 0.572 |
| v2 (4가지 전략) | **0.8975** | **0.883** | **0.712** |

### 5-2. 결과 해석
- Private +0.083 상승: 전략이 실제 일반화 성능을 개선했음을 증명
- Public과 Private 간 큰 차이: test set 분포가 train과 상당히 다름
  → 이것이 1등이 pseudo labeling을 사용한 이유
- 단일 fold만으로 0.883: 5-fold 앙상블 시 추가 상승 여지 존재

## Phase 6: 상위 솔루션 분석 — "천장은 어디인가"

### 6-1. 1등 솔루션 (Private 0.936)과의 Gap 분석

**우리가 이미 잘한 것:**
- 순서형 접근 (CORAL) — 1등도 회귀(SmoothL1) 사용
- 데이터 불균형 처리
- 적절한 augmentation

**1등이 추가로 한 것 (Gap 원인):**
- 외부 데이터 10배 확보 (2015 DR + IDRiD + Messidor)
- Pseudo labeling으로 test 분포 반영
- 4종 아키텍처 8개 모델 앙상블
- 512×512 큰 입력
- GeM Pooling
- QWK threshold 수동 최적화

### 6-2. 80/20 법칙 적용
- 0.883까지는 기본기(전처리, Loss, augmentation)로 도달 가능
- 0.883 → 0.936 구간은 데이터 전략과 앙상블이 지배
- 즉, 모델 아키텍처 혁신이 아니라 **데이터 엔지니어링**이 최상위권의 핵심

## 전체 사고 프레임워크 요약

### 원칙 1: 파이프라인 먼저, 최적화 나중
작동하는 end-to-end를 먼저 만들고, 제출까지 완료한 뒤 개선.
처음부터 완벽한 코드를 짜려 하면 아무것도 제출하지 못함.

### 원칙 2: 데이터를 먼저 의심하라
성능 문제의 대부분은 모델이 아니라 데이터 파이프라인에 있음.
전처리 불일치, 라벨 오류, 정규화 누락이 가장 흔한 원인.

### 원칙 3: 평가지표가 전략을 결정한다
QWK라는 지표의 특성(순서형 페널티)이
Loss 함수 선택(CORAL/회귀)과 후처리(threshold 최적화)를 직접 결정함.

### 원칙 4: 제약 조건 안에서 최선을 찾아라
무료 Colab T4라는 제약 → B3 + 300px + 체크포인트 전략.
무제한 GPU였다면 B5 + 512px + 5-fold가 가능했겠지만,
제약 안에서 우선순위를 정해 가장 효과적인 조합을 선택.

### 원칙 5: 천장을 알아야 방향이 보인다
1등 솔루션 분석으로 "데이터가 모자라다"는 진짜 병목을 파악.
모델만 바꿔서는 0.90을 넘기 어렵고,
외부 데이터 + pseudo labeling이 다음 단계의 열쇠.
